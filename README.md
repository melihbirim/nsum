# nsum: 3SUM, four ways

Count/find triples `a + b + c == target` in an integer array, with four algorithms
benchmarked against the Sedgewick/Wayne **algs4** `ThreeSum` inputs. One question:
which wins, and when?

There is **no** sub-quadratic general 3SUM (the 3SUM conjecture). FFT is the only
escape, and only for bounded integer ranges.

## Which one do I use?

- **Two-pointer, O(n²): the default.** Any integers, counting or listing. Fastest
  counter measured. Needs a sort (free in the noise).
- **Hash map, O(n²): niche.** Only when you can't sort. Same complexity, ~2.8× slower.
- **FFT, O(U log U): escape hatch.** Only when values sit in a small bounded range `U`
  *and* `n` is large. Decides *and* counts exactly. Wide range or small `n`: don't.

## Files

| file | what |
|---|---|
| `nsum.c` | two-pointer 3SUM to a target, prints triples |
| `nsum_simd.c` | NEON reference (O(n³), superseded by `nsum.c`; kept for the intrinsics) |
| `nsum_fft.c` | FFT decision: is target reachable? + one witness |
| `fft_count.c` | FFT exact counter, O(U log U) |
| `bench.c` | brute / hash / two-pointer counter, cross-validated + timed |
| `bench_all.sh` | builds all, runs the benchmark, emits markdown (CI uses it) |
| `*ints.txt` | algs4 `ThreeSum` inputs |

## Build & run

```sh
cc -O3 -o nsum nsum.c && ./nsum            # self-test, then: ./nsum 0 -1 0 1 2 -1 -4
cc -O3 -o bench bench.c && ./bench 8Kints.txt        # add --brute for the O(n³) reference
cc -O3 -o fft_count fft_count.c -lm && ./fft_count 8Kints.txt 0
./bench_all.sh                             # full benchmark, markdown out
```

Large inputs are gitignored; regenerate:

```sh
python3 -c "import random;print(' '.join(str(random.randint(-1000000,1000000)) for _ in range(64000)))" > 64Kints.txt
```

## Results

Counting zero-sum triples `i<j<k`. Counts cross-validated across methods and matching
algs4's published numbers (70, 528, 4039, 32074, 255181, 2052358, 16377796 for 1K–64K).

Times (seconds, Apple arm64, -O3):

| n | brute O(n³) | hash O(n²) | two-pointer O(n²) | FFT O(U log U) |
|---|---|---|---|---|
| 1K | 0.035 | 0.003 | 0.0013 | 0.45 |
| 8K | 16.22 | 0.230 | 0.083 | 0.48 |
| 32K | ~1040* | 4.262 | 1.367 | 0.49 |
| 64K | ~8300* | 17.40 | 5.539 | 0.45 |

`*` extrapolated (brute infeasible past 8K).

- Brute ×8 per doubling. Dead by 16K.
- Hash and two-pointer both n²; two-pointer ~2.8× faster (sequential access, no hashing).
- **FFT is flat in `n`** because its cost tracks the value range `U`, not `n`. On these
  inputs `U ≈ 2M`, so ~0.5s at every size. It overtakes two-pointer around 20–25K, and
  the gap only widens: at **10M** bounded values, two-pointer is ~36 h (extrapolated),
  FFT ~5.6 s.

## How FFT counts (not just decides)

Build the value histogram `P`. One self-convolution `conv2 = P*P` gives the pair-sum
spectrum. The exact triple count is

```
count = (S1 - 3A + 2B) / 6
```

where `S1 = Σ cnt[c]·conv2[T-c]` (ordered triples, indices independent) and `A`, `B` are
cheap O(U) corrections removing cases where two or three indices coincide (inclusion-
exclusion). All O(U log U). Precision caveat: double FFT rounds counts; for huge counts
switch to an integer NTT (validated exact through 64K here).

## Trust / no hallucinated numbers

Every claim here is executable and checked against something not generated by the author:

- **Cross-validation:** four independent algorithms must agree, or `bench` prints `MISMATCH`.
- **External ground truth:** counts match algs4's published reference.
- **Assertions:** each binary's `demo()` asserts known cases and aborts on failure.
- **CI runs it:** the release workflow executes `bench_all.sh` on a clean runner and writes
  the real output into the release notes. See the numbers in
  [releases](https://github.com/melihbirim/nsum/releases).

## Notes on the other two methods

**SIMD (`nsum_simd.c`).** NEON compares 4 `int32` per instruction (`vdupq_n_s32` broadcast,
`vld1q_s32` load, `vceqq_s32` compare, `vmaxvq_u32` reduce), ~4× on the inner scan. Dropped
anyway: it accelerated an O(n³) membership scan, and two-pointer's algorithmic win (~n/6×)
dwarfs a 4× constant. Lesson: pick the algorithm first, then vectorize the right one.

**Two-pointer duplicates.** On a match with `a[lo] != a[hi]`, count block-of-lows ×
block-of-highs and skip both; when the whole window is one value, pairs are `m(m-1)/2`.
